# 团队协作流程

建立团队协作规范，确保积木体系在多人协作中高效运转。

## 协作原则

1. **代码与积木同步提交**：修改代码时必须同步更新积木
2. **积木更新纳入 Code Review**：Review 代码时检查积木是否更新
3. **冲突优先解决**：积木冲突优先级高于代码冲突
4. **定期维护**：每月 Review 一次积木质量

## 开发流程

### 1. 开发新功能

```
创建分支 → 写代码 → 创建/更新积木 → 提交 MR → Code Review → 合并
```

**关键点**：
- 代码和积木在同一个 MR 中
- MR 描述中说明影响的积木
- Review 时检查积木是否同步更新

### 2. 修改现有功能

```
创建分支 → 找到相关积木 → 修改代码 → 更新积木 → 提交 MR → Code Review → 合并
```

**关键点**：
- 先找到相关积木，理解现有流程
- 修改代码后立即更新积木
- 变更记录中追加本次修改

### 3. Bug 修复

```
创建分支 → 修复代码 → 判断是否需要更新积木 → 提交 MR → Code Review → 合并
```

**判断标准**：
- 如果 bug 是流程设计问题 → 更新积木
- 如果 bug 是实现问题 → 不更新积木

## MR 模板

在项目根目录创建 `.github/PULL_REQUEST_TEMPLATE.md`：

```markdown
## 变更内容
{描述你的变更}

## 影响的积木
- [ ] 无积木受影响
- [ ] 已更新以下积木：
  - [ ] {积木文件名}
- [ ] 已创建新积木：
  - [ ] {积木文件名}

## 测试
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 手动测试通过

## 检查清单
- [ ] 代码符合编码规范
- [ ] 积木文件已同步更新（如适用）
- [ ] 变更记录已追加
- [ ] 索引文件已更新（如新增积木）
```

## Code Review 检查清单

### 代码 Review

- [ ] 代码逻辑正确
- [ ] 符合编码规范
- [ ] 测试覆盖充分

### 积木 Review

- [ ] 积木文件已更新（如适用）
- [ ] 流程图与代码一致
- [ ] 节点逻辑准确
- [ ] 锚点正确
- [ ] 异常路径完整
- [ ] 变更记录已追加
- [ ] 索引已更新（如新增积木）

## 角色分工

### 开发者

- 修改代码时同步更新积木
- 创建新功能时创建新积木
- 提交 MR 时说明影响的积木

### Reviewer

- Review 代码时检查积木是否更新
- 检查积木质量（流程图、节点逻辑、锚点）
- 发现问题及时反馈

### Tech Lead

- 定期 Review 积木质量
- 推动团队遵守积木维护规范
- 解决积木相关的争议

## 常见问题

### Q: 开发者忘记更新积木怎么办？

A: Reviewer 在 Code Review 时指出，要求补充。

### Q: 积木质量不高怎么办？

A: 
1. Reviewer 提出具体的改进建议
2. Tech Lead 定期 Review，统一标准
3. 分享优秀积木案例，供参考

### Q: 团队成员不愿意维护积木怎么办？

A: 
1. Leader 以身作则
2. 展示积木的价值（新人上手快、问题排查快）
3. 纳入绩效考核（可选）
